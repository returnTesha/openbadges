# 실제 배지가 어떻게 생겼나

## 서명 전 (Unsigned Credential)

발급자가 배지를 만들면 이런 JSON이 나온다:

```json
{
  "@context": [
    "https://www.w3.org/ns/credentials/v2",
    "https://purl.imsglobal.org/spec/ob/v3p0/context-3.0.3.json"
  ],
  "id": "https://example.com/credentials/3732",
  "type": ["VerifiableCredential", "AchievementCredential"],
  "issuer": {
    "id": "did:web:example.com:issuers:123",
    "type": "Profile",
    "name": "한국폴리텍대학",
    "url": "https://www.kopo.ac.kr"
  },
  "issuanceDate": "2025-06-15T09:00:00Z",
  "expirationDate": "2027-06-15T09:00:00Z",
  "name": "AI 융합 전문가 과정 수료",
  "credentialSubject": {
    "type": "AchievementSubject",
    "identifier": [
      {
        "type": "IdentityObject",
        "hashed": true,
        "identityHash": "sha256$c7ef86405ba71b85acd8e2e95166c4b111448089f2e1599f42fe1bba46e865c5",
        "identityType": "emailAddress",
        "salt": "not-a-pepper"
      }
    ],
    "achievement": {
      "id": "https://example.com/achievements/ai-convergence",
      "type": "Achievement",
      "name": "AI 융합 전문가",
      "description": "인공지능 기초, 머신러닝, 딥러닝, 컴퓨터비전 전 과정을 이수하고 프로젝트를 완료한 자에게 수여",
      "criteria": {
        "narrative": "전 과목 B+ 이상 이수 및 최종 프로젝트 Pass"
      },
      "image": {
        "id": "https://example.com/badges/ai-convergence.png",
        "type": "Image"
      }
    }
  }
}
```

### 각 필드 설명

| 필드 | 의미 |
|------|------|
| `@context` | 이 JSON이 어떤 표준을 따르는지 (W3C VC + OB 3.0) |
| `id` | 이 credential의 고유 식별자 (URL) |
| `type` | VerifiableCredential + AchievementCredential |
| `issuer.id` | 발급자의 DID (did:web → HTTPS URL로 변환 가능) |
| `issuanceDate` | 발급일 |
| `expirationDate` | 만료일 (선택) |
| `credentialSubject` | 수령자 + 달성 내용 |
| `identityHash` | 수령자 이메일의 SHA-256 해시 (프라이버시 보호) |
| `achievement` | 배지 정의 (이름, 설명, 기준, 이미지) |

### 수령자 이메일 해싱

이메일을 그대로 넣으면 프라이버시 문제가 있으니, **해시**해서 넣는다:

```
email = "hong@example.com"
salt  = "not-a-pepper"

SHA-256("hong@example.com" + "not-a-pepper")
  = "c7ef86405ba71b85acd8e2e95166c4b111448089f2e1599f42fe1bba46e865c5"

→ identityHash = "sha256$c7ef86405ba71b85..."
```

검증할 때 수령자가 자기 이메일을 알려주면, 같은 방식으로 해시해서 일치하는지 확인.

## 서명 후 (Signed Credential) — 이게 최종 배지

위 문서에 `proof` 블록이 추가된다:

```json
{
  "@context": [...],
  "id": "https://example.com/credentials/3732",
  "type": ["VerifiableCredential", "AchievementCredential"],
  "issuer": { ... },
  "credentialSubject": { ... },

  "proof": {
    "type": "DataIntegrityProof",
    "created": "2025-06-15T09:00:00Z",
    "verificationMethod": "did:web:example.com:issuers:123#key-1",
    "cryptosuite": "eddsa-rdfc-2022",
    "proofPurpose": "assertionMethod",
    "proofValue": "z58DAdFfa9SkqZMVPxAQpic7ndTn23CLz2wriRCPM2mUhiPXA2hEE4kP2qDnbRCFxXjEQh8B2dPYsNgr5iFFjSoU"
  }
}
```

### proof 필드 설명

| 필드 | 의미 |
|------|------|
| `type` | DataIntegrityProof (W3C 표준 서명 형식) |
| `created` | 서명 생성 시점 |
| `verificationMethod` | 검증에 쓸 공개키의 위치 (DID URL + key fragment) |
| `cryptosuite` | 사용한 암호 스위트: `eddsa-rdfc-2022` |
| `proofPurpose` | 서명 목적: `assertionMethod` (주장/발급 용도) |
| `proofValue` | **서명값 본체** (base58-btc 인코딩된 Ed25519 서명) |

## 검증 시나리오

### 시나리오 1: 정상 배지
```
검증자: "이 배지 진짜야?"
  → verificationMethod에서 공개키 가져옴
  → 문서 정규화 + 해시
  → Ed25519 Verify(공개키, 해시, proofValue)
  → ✅ true — "네, 한국폴리텍대학이 발급한 진짜 배지입니다"
```

### 시나리오 2: 위조된 배지 (내용 변조)
```
나쁜사람이 "achievement.name"을 "박사 학위"로 바꿈
  → 문서 정규화 + 해시 → 해시값이 달라짐
  → Ed25519 Verify → ❌ false — "서명이 일치하지 않습니다"
```

### 시나리오 3: 가짜 서명
```
나쁜사람이 자기 키로 proofValue를 새로 만듦
  → verificationMethod는 원래 발급자를 가리킴
  → 원래 발급자의 공개키로 검증
  → ❌ false — "서명이 일치하지 않습니다"
```

### 시나리오 4: 발급자 서버 다운
```
발급 기관 서버가 죽음
  → BUT 공개키가 DID 문서나 캐시에 남아있으면
  → 여전히 검증 가능 ✅
  → 이게 OB 2.0 대비 OB 3.0의 핵심 장점
```

### 시나리오 5: 취소된 배지
```
발급자가 취소 목록에 이 credential ID를 등록
  → 서명 자체는 유효 (true)
  → BUT 취소 상태 체크 → "취소됨"
  → 결과: "서명은 진짜지만 취소된 배지입니다"
```

## DID (Decentralized Identifier)란?

배지에서 발급자를 식별하는 방법이다.

### did:web (우리가 쓸 방식)

```
did:web:example.com:issuers:123

→ 이것은 곧 아래 URL을 의미:
   https://example.com/issuers/123/did.json

→ 거기에 이런 DID Document가 있어야 함:
{
  "id": "did:web:example.com:issuers:123",
  "verificationMethod": [{
    "id": "did:web:example.com:issuers:123#key-1",
    "type": "Multikey",
    "controller": "did:web:example.com:issuers:123",
    "publicKeyMultibase": "z6Mkf5rGMoatrSj1f..."  ← Ed25519 공개키
  }]
}
```

검증자는:
1. DID에서 URL 추출
2. HTTP GET으로 DID Document 가져옴
3. 거기서 공개키 추출
4. 공개키로 서명 검증

### did:key (간단 버전)

```
did:key:z6Mkf5rGMoatrSj1f4CyvuHBeXJELe9RPdzo2PKGNCKVtZxP

→ DID 자체에 공개키가 내장되어 있음
→ 별도 서버 조회 불필요
→ 대신 키 변경이 불가능
```
