# 핵심: 서명과 검증 알고리즘

Open Badges 3.0의 **진짜 핵심**은 "특정 데이터를 암호화"하는 게 아니라,
**"이 배지를 정말 이 발급자가 만들었다"는 걸 수학적으로 증명**하는 것이다.

> 암호화(encryption) ≠ 서명(signing)
> - 암호화: 내용을 숨긴다 (남이 못 읽게)
> - 서명: 내용은 공개하되, **누가 만들었는지 증명**한다 (위조 방지)

Open Badges는 **서명(signing)**을 사용한다.

---

## 1단계: 키 쌍 생성 (한 번만)

```
발급자가 처음에 키 쌍을 만든다:

  ┌─────────────────┐     ┌─────────────────┐
  │   비밀키 (SK)    │     │   공개키 (PK)    │
  │  Private Key     │     │  Public Key      │
  │                 │     │                 │
  │  32 bytes       │     │  32 bytes       │
  │  발급자만 보관   │     │  세상에 공개     │
  └─────────────────┘     └─────────────────┘

  알고리즘: Ed25519 (타원곡선 edwards25519)
```

- **비밀키**: 발급자만 안전하게 보관. 이걸로 서명을 만든다
- **공개키**: 누구나 볼 수 있게 공개. 이걸로 서명을 검증한다
- Ed25519는 RSA보다 키가 작고(32bytes) 빠르고 안전하다

---

## 2단계: 배지 발급 (서명 생성)

배지를 발급할 때 일어나는 일을 단계별로:

### Step 1: 배지 JSON-LD 문서 만들기

```json
{
  "@context": [
    "https://www.w3.org/ns/credentials/v2",
    "https://purl.imsglobal.org/spec/ob/v3p0/context-3.0.3.json"
  ],
  "type": ["VerifiableCredential", "AchievementCredential"],
  "issuer": {
    "id": "did:web:example.com:issuers:123",
    "type": "Profile",
    "name": "서울대학교"
  },
  "issuanceDate": "2025-01-15T09:00:00Z",
  "credentialSubject": {
    "type": "AchievementSubject",
    "achievement": {
      "type": "Achievement",
      "name": "데이터 사이언스 수료",
      "description": "DS 과정 전 과목 이수",
      "criteria": {
        "narrative": "전 과목 B+ 이상 이수"
      }
    }
  }
}
```

### Step 2: 정규화 (Canonicalization) — RDFC-1.0

JSON은 같은 데이터도 여러 방식으로 표현 가능하다:
```json
{"name": "홍길동", "age": 30}
{"age": 30, "name": "홍길동"}     ← 같은 데이터, 다른 문자열
```

서명은 **바이트 단위로 정확해야** 하니까, 먼저 **정규화**를 한다.

```
원본 JSON-LD
    ↓
[RDF 변환] — JSON-LD를 RDF 트리플(주어-술어-목적어)로 변환
    ↓
[RDFC-1.0 정규화] — 트리플을 알파벳 순 정렬, 공백 통일 등
    ↓
정규화된 문자열 (항상 동일한 바이트)
```

**핵심**: 같은 데이터면 누가 어디서 정규화해도 **바이트 단위로 동일한 결과**가 나온다.

### Step 3: 해싱 (SHA-256)

```
정규화된 문서     →  SHA-256  →  32 bytes (문서 해시)
정규화된 proof옵션 →  SHA-256  →  32 bytes (proof 해시)

두 해시를 이어붙임:
[proof 해시 32 bytes] + [문서 해시 32 bytes] = 64 bytes
```

왜 두 개를 따로 해시하나?
- **문서 해시**: 배지 내용이 변조되지 않았음을 보장
- **proof 해시**: 서명 시점, 목적 등 메타데이터가 변조되지 않았음을 보장

### Step 4: 서명 (Ed25519)

```
         64 bytes 해시 데이터
              ↓
  ┌─────────────────────┐
  │  Ed25519 Sign(SK)   │  ← 비밀키로 서명
  └─────────────────────┘
              ↓
         64 bytes 서명값
              ↓
     base58-btc 인코딩
              ↓
    "z3fFfxgwjy1..."  (proofValue)
```

### Step 5: proof를 붙여서 완성

```json
{
  "@context": [...],
  "type": ["VerifiableCredential", "AchievementCredential"],
  "issuer": { ... },
  "credentialSubject": { ... },
  "proof": {
    "type": "DataIntegrityProof",
    "created": "2025-01-15T09:00:00Z",
    "verificationMethod": "did:web:example.com:issuers:123#key-1",
    "cryptosuite": "eddsa-rdfc-2022",
    "proofPurpose": "assertionMethod",
    "proofValue": "z3fFfxgwjy1oPpjT7sGrMGRUQ2v..."
  }
}
```

## 3단계: 배지 검증 (서명 검증)

누군가가 이 배지를 받았을 때, **진짜인지 확인**하는 과정:

```
서명된 배지 JSON
    ↓
① proof 분리 (proofValue 추출)
    ↓
② 나머지 문서를 동일하게 정규화 + 해싱 (Step 2~3 반복)
    ↓
③ proofValue를 base58-btc 디코딩 → 64 bytes 서명값
    ↓
④ 발급자의 공개키(PK) 가져오기
   (verificationMethod URL에서 DID 문서 조회)
    ↓
⑤ Ed25519 Verify(PK, 해시, 서명값) → true/false
```

```
  ┌───────────────────────────────────────────┐
  │           Ed25519 Verify                  │
  │                                           │
  │  입력:                                    │
  │    - 공개키 (32 bytes)                    │
  │    - 해시 데이터 (64 bytes)               │
  │    - 서명값 (64 bytes)                    │
  │                                           │
  │  출력:                                    │
  │    - true  → 진짜 (이 발급자가 서명함)     │
  │    - false → 가짜 (위조 or 변조됨)        │
  └───────────────────────────────────────────┘
```

**핵심 포인트**:
- 비밀키 없이는 올바른 서명을 **절대 만들 수 없다**
- 문서 내용이 1비트라도 바뀌면 해시가 달라져서 검증 **실패**
- 발급자 서버가 죽어도, 공개키만 어딘가에 남아있으면 **검증 가능**

## 전체 흐름 요약

```
[발급]                              [검증]

배지 JSON                          서명된 배지
   ↓                                  ↓
정규화 (RDFC-1.0)                  proof 분리
   ↓                                  ↓
SHA-256 해시                       정규화 + SHA-256 해시
   ↓                                  ↓
Ed25519 Sign(비밀키)               Ed25519 Verify(공개키)
   ↓                                  ↓
proofValue 생성                    true / false
   ↓
proof 붙여서 완성
```

## 왜 Ed25519인가?

| 비교 | RSA-2048 | ECDSA P-256 | **Ed25519** |
|------|----------|-------------|-------------|
| 키 크기 | 256 bytes | 32 bytes | **32 bytes** |
| 서명 크기 | 256 bytes | 64 bytes | **64 bytes** |
| 서명 속도 | 느림 | 보통 | **빠름** |
| 검증 속도 | 빠름 | 보통 | **빠름** |
| 사이드 채널 공격 | 취약 | 주의 필요 | **내성 있음** |

1EdTech가 `eddsa-rdfc-2022`를 **필수 지원**으로 지정한 이유:
- 작고 빠르고 안전
- 구현이 단순해서 버그 가능성 낮음
- NIST FIPS-186-5에서 공식 승인

## 참고: JWT 방식과의 차이

OB 3.0은 DataIntegrityProof 외에 **JWT 방식**도 지원한다:

| 구분 | DataIntegrityProof | JWT |
|------|-------------------|-----|
| 서명 알고리즘 | Ed25519 (eddsa-rdfc-2022) | RS256 (RSA + SHA-256) |
| 데이터 형태 | JSON-LD + 내장 proof | Header.Payload.Signature |
| 정규화 | RDFC-1.0 (RDF 기반) | 불필요 (JWT 구조가 고정) |
| 다중 서명 | 가능 (proof 배열) | 불가 |
| 생태계 | W3C VC, DID | OAuth, 기존 웹 인프라 |

우리 프로젝트는 **DataIntegrityProof (Ed25519)** 를 기본으로 구현하되,
JWT도 추후 지원하는 방향으로 간다.
